# v0.9 报告 — listen `--backend auto`（first-utterance probe）

> 目标：让 `asrctl listen --backend auto` 也能工作——第一段说话探针定语言，
> 整个 session stick 选定的后端。
> 状态：✅ 通过（2026-05-02）。

## 用户体验

```sh
$ asrctl listen --backend auto -v
auto: probing first utterance…
（用户说话）
auto: detected language='zh' → backend=qwen3 (sticking for session)
backend:         qwen3
今天天气真不错
（继续说）
我们去公园散步吧
^C
stopped after 2 utterance(s)
```

第一段触发探针（额外 ~300 ms），后续段直接走选中后端正常推理。**整个 session 只探一次**。

## 设计决策

### 为什么选 first-utterance probe（不选 per-utterance）

per-utterance 重探每段加 50-100 ms tiny 推理 + 后端切换可能导致重新加载模型（qwen3 ↔ whisper 各自 1.5 GB / 547 MB）。在实时麦克风场景几乎不可接受。

first-utterance 假设：**用户在一个 session 里大概率讲同一种语言**。这个假设对 99 % 的用户场景成立。少数 code-switching 用户可以显式指定后端。

### 为什么 auto + --partial 拒绝

partial 路径需要在 session 启动时就有 `WhisperSession` 加载好（含 partial_ctx 的 tiny 模型）。auto 模式下我们到第一段才知道是不是 whisper，无法预先建好。强行支持有两条路：
1. 总是先加载 tiny + whisper-large-v3-turbo + qwen3 三份 → 太重
2. 等第一段后再开 partial → 第一段错过了 partial 体验

都不优雅。v0.9 选择硬拒绝 + 清晰错误提示，留给用户：
```
error: --backend auto is not compatible with --partial
       Pick a backend explicitly (qwen3 or whisper) when using --partial.
```

### 为什么 probe 完后第一段也喂回 onSegment

直观：用户已经说完了第一句话，他期望听到的是这句话的转录而非"探针完毕，请继续说"。所以探针内部 dupe 出 first segment 的 PCM，session 加载好之后用普通的 onSegment 路径转录这一段，与后续段无差别。

## 实现要点

```zig
// listen():
const choice = parseBackend(args.backend);

// VAD detector + audio capture 先就绪（与 backend 选择无关）
var detector = vad.Detector.init(...);
const capture = audio.Capture.start(...);

// auto 模式：probe 第一段
var first_segment_owned: ?[]f32 = null;
const backend: asr.Backend = switch (choice) {
    .real => |b| b,
    .auto => blk: {
        const samples = try probeFirstUtterance(allocator, io, &detector, capture)
            orelse return 0; // Ctrl-C
        first_segment_owned = samples;
        const lang = try asr.detectLanguage(allocator, tiny_path, samples);
        break :blk backendForLanguage(lang);
    },
};

// 真 Session 加载（现在知道 backend 了）
var session = try asr.Session.open(...);

// 把第一段转录一遍（如果是 auto 模式 probed 出来的）
if (first_segment_owned) |samples| {
    onSegment(&ctx, samples) catch ...;
}

// 主循环
while (running) { ... }
```

## 性能

probe 总开销（第一段说完后到看到文字）：
- tiny load + Metal JIT: ~500 ms（cold；warm cache 后 ~50 ms）
- detectLanguage 推理（5 s 探针）: ~50 ms
- 真 backend session.open：~500-1000 ms（qwen3 1.5 GB or whisper 547 MB）
- 第一段实际转录：100-500 ms

总计 cold start 第一段感知延迟 ~1.5-2 s。后续段恢复正常 ~300-700 ms。

vs 显式 backend：cold start 第一段差不多耗时（都要 load 真 backend），多出来的 ~500 ms 是 tiny load + 探针。trade-off 接受：用户不用想后端。

## 已知 limits

- **first-utterance 假设**：整个 session 同一语言。中英混杂会被识别为主导语言后 stick。用户切语言：要重启 listen。
- **不支持 --partial**：见上面设计决策章节。
- **probe 期间 user 视角无反馈**：探针中显示 "auto: probing first utterance…"（仅 verbose 下），其他模式下 silent。可能让用户疑心程序卡了——v0.10 可以加个 spinner。
- **first segment 时延 +500 ms**：tiny 模型加载 + 探针推理。后续段不受影响。
