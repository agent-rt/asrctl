# v0.2 报告 — 实时麦克风 ASR

> 目标：`asrctl listen` 实时麦克风 → 文本流。
> 状态：✅ 通过（2026-05-02）。

## 端到端

```sh
$ asrctl listen
（说话）"hello world"
（停顿 0.6s）→ "hello world\n" 打印到 stdout
（继续）"how are you"
（停顿）→ "how are you\n"
^C
```

## 架构

```
CoreAudio AudioQueue (16 kHz mono f32 PCM)
    ↓ callback thread (locks SpinLock, appends to ring)
ring buffer
    ↓ main loop (50 ms drain cadence)
vad.Detector (RMS + silence-duration FSM)
    ↓ on_segment callback per utterance
asr.Session.transcribePCM()  (model loaded once at listen start)
    ↓
mtmd_bitmap_init_from_audio + mtmd_helper_eval_chunks + greedy sample
    ↓
parseOutput → "<asr_text>" tag split → stdout / -o file
```

## 关键决策

### Qwen3-ASR + VAD 分段（路线 A），未切 whisper.cpp

REQ.md §10 留过决策点：流式不天然支持 → 是否换 whisper.cpp？最终选 A，理由：

1. Qwen3-ASR 整段质量已验证（M0/M2 结果逐字一致），切 whisper.cpp 要重新训练用户对识别效果的预期。
2. VAD-segmented 模式延迟 = silence_ms（默认 600ms）+ 模型推理（~0.3-0.6s），用户感受 ~1s。能用。
3. 完全复用 v0.1 的 mtmd 调用代码，新增的全是「audio capture + VAD」薄层。
4. 真要 streaming partial words，路线 B（whisper.cpp）成本更大，留给将来真的需要时。

### 直接 `extern` 声明 AudioQueue API，不走 translate-c

把 `<AudioToolbox/AudioQueue.h>` 加到 `c.h` → translate-c 拉进 mach_msg.h → mach_msg_descriptor_t 等 union 类型 sizing 失败（14 errors）。Apple 的 ObjC blocks `^` 语法也不支持。

解决：在 `audio.zig` 里手写 `extern "c" fn AudioQueueNewInput(...)` 等约 8 个声明 + AudioStreamBasicDescription 等 struct。C ABI 稳定，不会 break。

### SpinLock 替代 std.Thread.Mutex

Zig 0.16 的 `std.Thread.Mutex` 没了，`std.Io.Mutex` 要 io（CoreAudio 回调线程没 io 引用，传不过去）。回调和主循环各拿锁的时间是微秒级，spinlock + cmpxchg 完全够。

### `mtmd_bitmap_init_from_audio` 路径验证（v0.2.0 spike）

把 transcribe 命令也切到 PCM 路径（`wav.decodeFile` → `Session.transcribePCM`）做对照测试：4 个样本（EN/ZH × short/long）输出与原 file 路径完全一致。这同时为 listen 路径背书——同一个 `transcribePCM` 收到任何来源的 f32 PCM 都能跑。

## 踩到的 Zig 0.16 坑（追加 M3 列表）

| 旧 API | 0.16 替换 |
| --- | --- |
| `std.Thread.sleep` | 没了，用 `extern "c" fn usleep(u32) c_int` |
| `std.Thread.Mutex` | 没了，自己写 atomic SpinLock 或用 `std.Io.Mutex` |
| `std.posix.nanosleep` | 没了 |
| `c_int` 作 sighandler 参数类型 | 必须用 `std.posix.SIG` |
| `file.writerStreamingPositional` | 没了，用 `file.writePositionalAll(io, bytes, offset)` |

## 已知 limits（留给 v0.3+）

- **VAD 能量阈值是固定值**（默认 0.012，约 -38 dBFS）：嘈杂环境会误触，安静环境会漏。silero-VAD 比能量法好得多但依赖 ONNX Runtime，留给真有用户反馈再决定是否引入。
- **不输出 partial 结果**：必须等 silence 触发 segment。要 OpenAI Whisper 风格的 streaming partial，需要换 whisper.cpp 或自己做 sliding window。
- **mic 设备硬选默认**：`--device <name>` flag 还没接，用户切 BlackHole/AirPods 要改系统默认输入。
- **macOS 首次跑要批 mic permission**：CoreAudio 弹窗。CI 不能验。
- **首次 utterance 慢**：模型 load + Metal kernel JIT 还在主线程发生（约 1s）。listen 启动时 ggml log 可见。M0 spike 报告里的 warmup 缓存优化未做。

## 决定

v0.2 收。下一步候选：
- 打 v0.2.0 tag 触发 release。
- 真的 silero VAD（ONNX）替能量 VAD。
- 设备选择 `--device`。
- partial result 流式（如果需要切模型）。
