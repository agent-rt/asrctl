# M5.5 — Vendor 静态化可行性 Spike

> 目标：弄清把 llama.cpp + ggml + mtmd 编进 asrctl 单二进制需要做什么，以及哪些是真阻塞。
> 状态：✅ 可行性确认，未实施。M5.5 实施工作量大，本次会话不做，留给独立 sprint。

## 阻塞 / 不阻塞验证

### Metal Toolchain → ❌ 不阻塞

```
xcodebuild -downloadComponent MetalToolchain
```

实测：自动下载 ~688 MB 的 dmg → 装到 `/System/Library/AssetsV2/...`。30 秒完成，无交互。装完 `xcrun metal --version` 直接可用。

文档需求：在 README 加一句 "vendor build 需要先跑这条命令"。

### Brew 静态库 → ❌ 不可用

```
$ find /opt/homebrew/Cellar/{ggml,llama.cpp} -name '*.a'
# 空
```

brew bottle 只有 `.dylib`，没有 `.a`。**意味着不能简单 `linkSystemLibrary` 静态链 brew 装的库**，必须从源码编译。

### upstream 源码作 zig 依赖 → ✅ 可行

```
zig fetch --save=llama_cpp git+https://github.com/ggml-org/llama.cpp
# → resolved to commit c3c15053... (主线滚动)
# → 写入 build.zig.zon
```

upstream llama.cpp 没有 `build.zig`，只有 CMakeLists。但 `zig fetch` 仍然可以把整个仓库当成"源码包"拉下来，build.zig 里手动 `b.dependency("llama_cpp", .{}).path("...")` 引用任意子目录文件。

这正是 `diogok/llama.cpp.zig` 的做法：~600 行 build.zig 列出所有 `.c` / `.cpp` 源文件、include path、cflags，再用 Metal toolchain 把 `.metal` 着色器编成 `.metallib`。

## 完整实施需要做的事

按工作量从低到高：

1. **`build.zig.zon`**：把 upstream llama.cpp pin 到一个稳定 commit（建议 8990 对应的 tag），加 hash。已演练，10 分钟工作量。

2. **构建 ggml 静态库**：扫 upstream `ggml/src/*.c|cpp`、`ggml/src/ggml-cpu/*.c`、`ggml/src/ggml-blas/*.cpp`、`ggml/src/ggml-metal/*.m`，加 cflags（`-DGGML_USE_METAL`、`-DGGML_USE_BLAS`、`-DNDEBUG`、`-D_GNU_SOURCE` for Linux），include `ggml/include` 和 `ggml/src`。约 30+ C/C++/Objective-C 文件。**主要工作量**。

3. **编译 Metal shaders**：`ggml/src/ggml-metal/ggml-metal.metal` → `default.metallib`。Zig build.zig 用 `b.addSystemCommand(&.{"xcrun", "metal", ...})`。需要也运行时能找到 metallib——两条路径：
   - (a) 把 `default.metallib` 二进制放到 `zig-out/bin/` 旁边。**简单但不是单二进制**。
   - (b) `@embedFile` 把 metallib 字节码内嵌进 binary，运行时 `mtl_library_init_with_data` 加载。需要 patch ggml-metal 的初始化逻辑。**真单二进制但有侵入**。

4. **构建 llama 静态库**：upstream `src/*.cpp` + `ggml-base` 链接。

5. **构建 mtmd 静态库**：upstream `tools/mtmd/*.cpp`，依赖 llama 和 ggml。还要拉进 stb_image / miniaudio 单头依赖。

6. **build-info.cpp 注入**：upstream 期待一个生成的 build-info.cpp 提供 commit/build-time 字符串。`b.addWriteFile` 临时生成。

7. **Backend register 改造**：当前代码用 `ggml_backend_load_all_from_path` 动态扫 libexec。静态构建里没有 .so，要改成调用 `ggml_backend_metal_init` / `ggml_backend_cpu_init` 等显式注册函数（这些在 ggml-base 里）。`asr.zig` 的初始化序列要改一处。

8. **链接配置**：macOS framework `Metal` / `Foundation` / `QuartzCore` / `Accelerate`，`linkLibCpp`，正确的 rpath/install_name 处理。

9. **二进制大小验证**：预期 ~80–150 MB。REQ.md §5 目标 ≤50 MB 在静态构建下不现实，需要修订（diogok 的同样产物大概也在这个量级）。

10. **打包 metallib**：如果走 3(a) 路径，分发要带 metallib 文件；走 3(b) 内嵌。

## 估算

参考 `diogok/llama.cpp.zig` 的 build.zig 是 ~600 行。即使复制改造，也要：
- 1–2 天熟悉 ggml 的 cmake 树和实际编译命令
- 1 天写 + debug build.zig
- 0.5 天处理 Metal shader + backend register
- 0.5 天处理 ABI / link 错误

总计 **3–4 天**，且大概率会遇到至少一个 upstream 内部 cmake 用了 zig 不直接支持的特性（比如 `add_compile_definitions(...)` 的 generator expression）需要手动展开。

## 当前 v0.1 取舍

继续用 brew dylib 链接。代价：
- 用户 `brew install llama.cpp ggml` 是前置（README 已写明）。
- 二进制不是真单文件，跨 mac 分发要么 brew 安装要么打包 dylib。

收益：**M5 收尾按时**，静态化作为 v0.2 / v1.0 独立 sprint。
