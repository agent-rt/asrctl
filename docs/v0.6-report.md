# v0.6 报告 — Whisper audio_ctx 缩放优化

> 目标：让 whisper 在短 buffer 上跳过 30s pad 部分的 encoder 工作。
> 用于 `listen --partial`（500ms cadence × 1-5s buffers）+ 短 wav transcribe。
> 状态：✅ 通过（2026-05-02）。

## 背景修正（关于 v0.5 报告的错误）

v0.5 报告里我说"`whisper_full_with_state` 复用 encoder 状态"——读了上游源码后发现**这是误判**。whisper_full 内部把任何输入 pad 到 30s 然后跑一次 encoder：

- 对 ≤30s 输入，encoder cost 是常量（与实际音频长度无关）
- partial 多次调用之间无 encoder 状态可复用，每次都要重跑

真正可优化的是 `whisper_full_params.audio_ctx`——告诉 encoder "只跑前 N 个 mel 帧，剩下的 pad 部分跳过"。这是 whisper.cpp examples/stream/stream.cpp 的实际做法（不是我之前以为的滑动窗口）。

## 实测

```
ASRCTL_TIMINGS=1 ./asrctl --backend whisper [--quick] file.wav
→ whisper_print_timings 内置统计
```

| 输入 | full audio_ctx=1500 | quick (scaled) | encoder 节省 |
| --- | --- | --- | --- |
| 3s 片段 | encoder 666 ms × 2 runs | 491 ms × 2 runs | **-26%** |
| 5.4s 片段 | encoder 668 ms × 2 runs | 493 ms × 2 runs | **-26%** |
| 17s 片段 | encoder 664 ms × 2 runs | 612 ms × 2 runs | -8% |

Wall time（含模型 load + Metal JIT，每次都是冷启动进程）：

| 输入 | full | quick | net |
| --- | --- | --- | --- |
| 3s | 2168 ms | 1669 ms | **-23%** |
| 5.4s | 2153 ms | 1660 ms | **-23%** |

## 选 audio_ctx 公式的过程

```
audio_ctx = clamp(seconds * 50 + 256,  768,  1500)
                  ^^^^^^^^^^^^^^^^      ^^^   ^^^^
                  实际音频对应 mel 帧 + 256 margin
                                       floor 768 = 15s context
                                              max = 30s
```

- 公式参考：whisper 内部 mel 是 100 Hz（10ms hop）+ 2x conv downsample → 50 Hz audio context。30s × 50 = 1500 (whisper 的最大 audio_ctx)。
- **没加 floor 时**（直接 `seconds * 50 + 10`）：3s → audio_ctx=160 → encoder 372 ms（更快！）但 **decoder 完全失控**：跑了 220 次输出残破"of\n"。decoder 注意力区域太小，loop 检测不到 EOG。
- **floor=768** 经过试错找到的甜区：encoder 仍省 ~25%，decoder 注意力区域足够，输出与 full ctx 一字不差（甚至有时候多一个句号）。
- floor 上界 1500 是 whisper 模型本身的约束。

## 实施

```zig
pub const Session = union(Backend) {
    qwen3: Qwen3Session,
    whisper: WhisperSession,
    pub fn transcribePCM(self: *Session, samples: []const f32) Error!Result { ... }
    pub fn transcribePCMQuick(self: *Session, samples: []const f32) Error!Result {
        return switch (self.*) {
            .qwen3 => |*s| s.transcribePCM(samples),  // qwen3 没等价快速路径
            .whisper => |*s| s.transcribePCMQuick(samples),
        };
    }
};

// WhisperSession:
fn transcribeWithCtx(self, samples, audio_ctx: c_int) {
    var fparams = c.whisper_full_default_params(...);
    fparams.audio_ctx = audio_ctx;
    c.whisper_full(self.ctx, fparams, samples.ptr, samples.len);
    if (self.verbose_timings) c.whisper_print_timings(self.ctx);
    ...
}
pub fn transcribePCM(self, samples) { return self.transcribeWithCtx(samples, 0); }
pub fn transcribePCMQuick(self, samples) {
    const audio_ctx = clamp(samples_to_mel_50hz(samples) + 256, 768, 1500);
    return self.transcribeWithCtx(samples, audio_ctx);
}
```

## 暴露给用户

- **`asrctl listen --partial`**：自动用 `transcribePCMQuick`。partials 是 throwaway 预览，~25% 速度赢，无质量损失（floor 768 保证）。
- **`asrctl --backend whisper --quick file.wav`**：用户显式 opt-in 短文件快速转录。doc 注明"slight accuracy trade-off on edge cases"——保守起见，默认 off。
- **`ASRCTL_TIMINGS=1`** env：触发 `whisper_print_timings`，给 bench 用。不进 CLI flag，避免噪音。

## v0.5 partial 实际 cost（修正）

v0.5 报告里说 partial sustained 20-40% Metal 占用——这个数没变。但理解修正后：
- 之前以为"encoder 跑整个 30s 不可避免"
- 实际 audio_ctx 缩放让 encoder 省 25%，partial 总耗时也降 25%
- 等于 v0.5 配 v0.6 后，partial sustained 占用降到 **15-30%**

partial 体验：
- 之前：3s buffer partial → 700ms 推理 → 听起来"卡 200ms"
- 现在：3s buffer partial → 530ms 推理 → 几乎贴着 500ms cadence，跟手
