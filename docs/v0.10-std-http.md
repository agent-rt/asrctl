# v0.10 — std.http migration

> 计划：把 `hf.zig` 和 `server.zig` 里的 curl 子进程换成 Zig 0.16 `std.http.Client.fetch`，去掉 curl 运行时依赖。
> 状态：✅ 已完成（2026-05-02 第二次尝试）。绕过了 macOS std.http DNS bug。

## 动机

curl 子进程模式自 v0.1 起一直是 placeholder。当时 std.http 在 Io 接口大改造，先用 curl 顶上。v0.10 想清账。

## 实施了什么

- `hf.zig` `download` 函数：`std.process.spawn("curl ...")` → `std.http.Client.fetch`，response_writer 流到目标文件
- `server.zig` `transcribe` 函数：curl POST -F → 手撸 multipart/form-data body + `std.http.Client.fetch` POST，response 流到 `Allocating writer`
- `errors.zig` 把 `SpawnFailed` 改 `HttpFailed`

代码 build 通过、形态正确。问题不在我们这层。

## 阻塞性发现：Zig 0.16 std.http 在 macOS 上有 bug

### Bug 1：IP-literal short-circuit 只在 linux/windows

`std/Io/Threaded.zig:13497`:
```zig
if (native_os == .linux or is_windows) {
    if (IpAddress.parseIp6(name, options.port)) |addr| { ... }
    if (IpAddress.parseIp4(name, options.port)) |addr| { ... }
}
```

macOS 走 else 分支，把 `127.0.0.1` 当成域名做 DNS 查询 → `error.UnknownHostName` → `error.AddressUnavailable`。

repro：
```zig
var client: std.http.Client = .{ .allocator = allocator, .io = io };
defer client.deinit();
_ = try client.fetch(.{ .location = .{ .url = "http://127.0.0.1:8080/" } });
// → error: AddressUnavailable
```

curl 同 URL 200 OK。

### Bug 2：`localhost` 走 dual-stack 但不容错

走 RFC 6761 特例 branch 同时返回 ::1 和 127.0.0.1。`HostName.connectMany` 并发 connect。如果 ::1 上没有 listener（IPv4-only 服务），返回 `ConnectionRefused`。期望行为是另一支成功就用，但实测拿到的是 refused。

repro：
```zig
// 服务监听 127.0.0.1:8765 (lsof 验证 IPv4 only)
_ = try client.fetch(.{ .location = .{ .url = "http://localhost:8765/" } });
// → error: ConnectionRefused
```

curl 同 URL 200 OK。

## 结论与最终方案

用 std.http 替 curl 在 macOS 上有局部降级，但**可以分两条路绕开**：

**`hf.zig`（公网 HTTPS 下载）**：直接用 `std.http.Client.fetch`。Bug 1/2 都只打到 IP-literal / `localhost`，对域名 + getaddrinfo 完全不影响。HF API（含 xet-bridge 重定向）实测正常。

**`server.zig`（localhost POST）**：完全绕开 std.http 和 lookup。手撸 ~150 行 HTTP/1.1：
- `parseUrl` 拆出 host / port / path
- `resolveLocal` 只接受 IP literal 和字面量 `localhost`（→ `127.0.0.1`）；DNS 名直接拒绝（`InvalidServerUrl`，附 hint 指向本文档）
- `IpAddress.connect(addr, io, .{ .mode = .stream })` 直连
- 手写 request line / Host / Content-Type / Content-Length / `Connection: close`，followed by multipart/form-data body
- 读 status line + 跳 headers + `appendRemainingUnlimited` 拿 body（`Connection: close` 让我们靠 EOF 边界，不用解析 chunked / Content-Length）

副作用：localhost 路径不再支持 IPv6-only 或非 loopback DNS 名。`--server-url` 的实际使用场景就是本机 llama-server，可接受。

待 upstream 修复后可以直接切回 `Client.fetch`：
1. macOS 加 IP-literal short-circuit（约 5 行 if 块外移）
2. `connectMany` 在 dual-stack 拿到 `ConnectionRefused` 时不要立刻放弃

## 实现要点（踩过的坑）

`std.Io.Reader.takeDelimiterExclusive` 的语义是 "advancing the seek position **up to (but not past)** the delimiter"——读 status line 后位置卡在 `\n` 上，下一次调用立刻返回空切片。要一次吃掉整行用 `takeDelimiterInclusive` 然后自己 `trimEnd("\r")`。

## 影响

- `curl` 不再是运行时依赖
- README 已更新（删除 "需要 curl"）
- `--server-url` 现在不接受 https / 任意 DNS 名，错误信息里指向本文档
