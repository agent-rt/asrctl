# v0.11 报告 — 实时音频路径优化（lock-free ring + wakeup pipe + adaptive silence cut）

> 目标：从音频回调到 VAD commit 的整条 listen pipeline 做三处实质改进。
> 状态：✅ 通过（2026-05-03）。

## 改了什么（A + B + C）

### A. Lock-free SPSC ring buffer (`audio.zig`)

**Before**: `SampleRing` 用 `std.ArrayList(f32)` + `SpinLock`。音频回调（real-time
线程）持锁后调 `appendSlice(allocator, samples)`——allocator 路径里任何一处卡住
（page fault / arena 锁 / realloc）都会把 RT 线程拽进慢路径，CoreAudio 文档明确
警告这是 dropout 来源。

**After**: 固定大小 `[]f32`（30 s × sample_rate ≈ 1.9 MB）启动时一次性 preallocate，
原子 `head` / `tail` 双计数器，无锁 SPSC。

- Producer（音频回调）：`load tail (acquire)` → `memcpy` 进 ring → `store head (release)`
- Consumer（listen loop）：`load head (acquire)` → `memcpy` 出 ring → `store tail (release)`
- 溢出策略：drop-newest，原子 `overflow` 计数器。30 s 容量足够吸收任何正常的消费者
  抖动；真触发 overflow 说明 consumer 卡了 30 s 以上，那是另一个 bug。`-v` 退出
  时打印丢弃量。

### B. Wakeup pipe (`audio.zig` + listen loop)

**Before**: listen loop `_ = usleep(50 * 1000)` 后再 drain。最坏情况在 idle → speaking
转场时多一个 50 ms 延迟下限，每秒 ~20 次空 wakeup。

**After**: pipe(2) 双端 `O_NONBLOCK`，回调成功写入 ring 后写 1 字节（EAGAIN 直接吞，
说明 wakeup 已经 pending）。Consumer 改成 `capture.waitForData(50)`——`poll()`
在音频到达瞬间唤醒，50 ms 超时保留 partial preview 节奏。

write(2) ≤ PIPE_BUF 是原子 syscall，real-time-safe。

### C. Adaptive silence cut（自适应句末切分，`vad.zig`）

**Before**: `Detector` 只有一条切分规则——`silence_ms = 600` ms 的 `voiced=false`
streak。`voiced=false` 阈值是 `P(speech) < silero_threshold = 0.5`，包含了大量
"不确定"帧（P=0.1–0.4）。结果：录音里的自然停顿（句间空隙、换气）一旦累积 600 ms
就触发 `safety_cut`，把一句话切成多段。

**After**: `Voicer.classify` 返回 `FrameClass { voiced, strong_silence }`。silero 的
`strong_silence` 在 `P ≤ 0.05` 时为 true（"确信安静"）。Detector 同时维护两个 run：

- `silence_run`：voiced=false 帧数 → `safety_cut @ 600 ms`（兜底）
- `strong_silence_run`：strong_silence=true 帧数 → `quick_cut @ silero_quick_silence_ms`

Energy backend 的 `strong_silence` 永远是 false，行为不变。

## bench 数据（`bench/results-vad-2026-05-03.txt`）

四份样本（en/zh × short/long），每份末尾 padding 1.5 s 纯静音。"speech end" 由
silero 自己的最后 voiced 帧定义（避免 buffer 末尾恰好落在音节中段，silero 的 LSTM
unwind 偏置数据）。表中 `latency_ms` = commit_frame − last_voiced_frame，单位 ms。

| sample   | quick=0（pre-v0.11） | quick=250（draft） | quick=400（v0.11 默认） |
|----------|--------------------|------------------|----------------------|
| en_short | **MID-SPEECH @ 2560 ms** | **MID-SPEECH @ 2752 ms** | 416 ms |
| zh_short | **MID-SPEECH @ 1920 ms** | 288 ms           | 448 ms |
| en_long  | **MID-SPEECH @ 3808 ms** | 256 ms           | 416 ms |
| zh_long  | **MID-SPEECH @ 1888 ms** | 256 ms           | 416 ms |

### 关键发现

**`quick=0`（pre-v0.11 行为）四份样本全部 MID-SPEECH cut**——不是新现象，是 600 ms
safety threshold 单独看 voiced=false 时的固有行为：录音里的句间停顿是 ambiguous
silence（silero P=0.1–0.4），累计 600 ms 就触发。结果是真实使用中"一句话被切成
两段"的体感。**这是 v0.11 之前就存在的 bug，但之前从没量化过。**

**`quick=250`（最初 draft）3/4 干净**：zh_short / en_long / zh_long 在真实句末后
256–288 ms commit。en_short 仍 MID-SPEECH——里面有一段 ~250–400 ms 长的 strong
silence（音节间隔），250 ms 阈值不够避开。

**`quick=400`（最终默认）4/4 干净**：所有样本在真实句末后 416–448 ms commit。比
draft 多 ~160 ms 句末延迟，换 4/4 而不是 3/4 的可靠分句。

### 决策

默认值从原来 draft 的 250 ms 改成 **400 ms**。理由：

- 真实使用中"一句话被切成多段"是用户立刻能察觉的 bug（看到两行输出、ASR 也对
  half-utterance 推理质量更差）；多 160 ms 句末等待用户能察觉但能接受
- 400 ms 仍显著好于 pre-v0.11 的"任何 600 ms 的 ambiguous silence 都切"——后者
  实际触发时机不可预测、在所有测试样本里都过早
- 想再激进可以传 `silero_quick_silence_ms = 250`（API 是 public 的）；用户实际
  反馈再调整默认值

## 用户可感知

- **句末 commit 不再被内部停顿打断**：原本 5–17 s 的连续讲话会被切成 2–4 段，现在
  是一段
- **句末延迟稳定在 ~416 ms**：之前是 1.9–3.8 s（极不稳定，取决于第一个内部停顿
  在哪），现在 416–448 ms，可预期
- **idle → speaking 起话瞬间反应更快**：从 ≤ 50 ms 抖动 → 立即（wakeup pipe）
- **重负载下不会随机丢音**：移除了 RT 线程上的 allocator 调用

## 没做（明确放弃）

- **WER bench**：v0.11 不动 ASR 模型，分句变化对 WER 的影响是间接的；想看的话
  v0.12 单独做
- **mp3 / m4a 输入**：用户明确说不做，现在核心是实时音频
- **真实房间噪音下 quick_cut 的有效性**：bench 用纯零静音，silero P 能掉到 ≤ 0.05；
  实际房间底噪可能让 P 长期 > 0.05，quick_cut 不触发，degrades 到 safety_cut（仍是
  600 ms，比 pre-v0.11 不更差）。这部分等真实使用反馈
