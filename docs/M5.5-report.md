# M5.5 报告 — Vendor 静态化

> 目标：单二进制，无 brew 运行时依赖。
> 状态：✅ 通过（2026-05-02），同一会话完成。

## 关键转折：cmake-shellout 取代手写 build.zig

Spike 阶段（[M5.5-vendor-spike.md](./M5.5-vendor-spike.md)）估算手写 build.zig 要 3-4 天（参考 diogok/llama.cpp.zig 的 ~600 行）。实际 1 小时搞完，因为换了思路：

> **让 cmake 干编译重活，build.zig 只负责 link。**

`b.addSystemCommand("cmake --build ...")` 输出 7 个 `.a` 文件，build.zig 通过 `addObjectFile` 静态链。一次性配置 vs 600 行手写 cmake-tree port，工作量差一个数量级。

代价：cmake 加入构建依赖（macOS 自带 Xcode CLT 已有，CI runner 也有，README 文档了）。

## cmake flags

```
-DBUILD_SHARED_LIBS=OFF        # 产出 .a
-DGGML_BACKEND_DL=OFF          # 不要插件机制
-DGGML_NATIVE=OFF              # 跨机器可分发，不用 -mcpu=native
-DGGML_METAL=ON                # 启用 Metal 后端
-DGGML_METAL_EMBED_LIBRARY=ON  # ✨ shader 源码内嵌为 C 字符串，运行时编译
-DGGML_BLAS=ON                 # Accelerate
-DGGML_OPENMP=OFF              # 没装 libomp
-DLLAMA_BUILD_TESTS=OFF
-DLLAMA_BUILD_EXAMPLES=OFF
-DLLAMA_BUILD_SERVER=OFF
-DLLAMA_BUILD_TOOLS=ON         # 仅为了编 mtmd 静态库（CMakeLists 把 mtmd 放在 tools/ 里）
-DLLAMA_CURL=OFF
```

`GGML_METAL_EMBED_LIBRARY=ON` 是关键：把 `.metal` shader 源码嵌进 `libggml-metal.a`，运行时由 macOS Metal driver 编译，**不需要 `xcrun metal` 工具链**。这同时解决了 M5.5b 和「单二进制不能带 metallib」的需求。

## Backend 自动注册

`-DGGML_BACKEND_DL=OFF` + `-DGGML_USE_METAL/BLAS/CPU` 的 cmake 路径让 `ggml-backend-reg.cpp` 内的 `ggml_backend_registry()` 构造函数在 static-init 时自动调用 `register_backend(ggml_backend_metal_reg())` 等。**不用显式 init 调用，不用 `ggml_backend_load_all_from_path`，asr.zig 干干净净**。

`backend.zig` 文件直接删除。

## 踩过的坑

1. **build.zig 路径打错**：`ggml/src/libggml-metal/libggml-metal.a` ≠ `ggml/src/ggml-metal/libggml-metal.a`，shell 报错信息很清晰一秒就修。
2. **`addObjectFile` 在 0.16 是 module 上的方法**：跟 `addLibraryPath` / `linkFramework` 一样，全部从 Compile 挪到 Module 上。
3. **link 顺序**：mtmd → llama → ggml → ggml-base + 三个 backend lib，这是 cmake 的 `target_link_libraries` 顺序的反向（链接器从前到后查未解析符号）。
4. **空跑日志混淆**：第一次失败后 `zig build` 第二次成功但显示 "cached" 没有打印任何路径，让我以为没产出。`--summary all --verbose` 看到 `install -C ... zig-out/bin/asrctl`。

## 收益

- ✅ 单二进制：用户只需 `brew install agent-rt/tap/asrctl`，不再级联拉 llama.cpp/ggml。
- ✅ 版本锁定：upstream commit pin 在 build.zig.zon，不会因 brew 升级造成 ABI 漂移（这正是之前 0.9.11 vs 0.10.1 撞 segfault 的根源）。
- ✅ CI 简化：不再装 brew dependencies，runner 自带 cmake + Xcode CLT 就够。
- ✅ 性能不变：静态 = 动态。
- 工程量从 spike 估的 3-4 天降到 1 小时，得益于换思路 cmake-shellout。
